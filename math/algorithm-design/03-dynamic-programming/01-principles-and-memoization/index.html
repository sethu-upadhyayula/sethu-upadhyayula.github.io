3.1 Principles and Memoisation

Dynamic programming works when a problem has two key structural properties. Identifying these properties is the first step in designing a DP algorithm.

Definition 3.1.1 — Optimal Substructure

A problem has optimal substructure if an optimal solution to the problem contains optimal solutions to its subproblems. Formally: if we can decompose an optimal solution into parts, each part is optimal for the corresponding subproblem. Proof technique: cut-and-paste — if any part were suboptimal, we could replace it with an optimal part, improving the overall solution (contradiction).

Definition 3.1.2 — Overlapping Subproblems

A problem has overlapping subproblems if a recursive algorithm revisits the same sub-problems many times. This is the key difference from divide and conquer, where subproblems are independent. DP stores each subproblem's solution once in a table.

Top-Down vs. Bottom-Up

Definition 3.1.3 — Memoisation (Top-Down)

Memoisation: write the natural recursive solution; add a cache (hash map or array). Before computing a subproblem, check if it is already cached. If so, return the cached value; otherwise compute, cache, and return. Pro: only computes subproblems actually needed. Con: function call overhead; risk of stack overflow for deep recursion.

Definition 3.1.4 — Tabulation (Bottom-Up)

Tabulation: fill a table in topological order of the subproblem DAG. Each cell is computed exactly once using already-filled cells. Pro: no recursion overhead; cache-friendly access patterns. Con: must compute all subproblems (no lazy evaluation).

Example 3.1.5 — Fibonacci Numbers

Naive recursion: $T(n) = T(n-1) + T(n-2) + O(1) = O(2^n)$. With memoisation or tabulation: each $F(k)$ computed once, $T(n) = O(n)$.

Space optimisation: only $F(k-1)$ and $F(k-2)$ are needed at each step, so $O(1)$ space suffices (rolling array).

Recipe for DP Design

Recipe 3.1.6 — DP Design Steps
  1. Define the state: what information characterises a subproblem? (e.g., index $i$, remaining capacity $w$, etc.)
  2. Write the recurrence: express the value of a state in terms of smaller states.
  3. Identify base cases: smallest states with known values.
  4. Determine computation order: ensure dependencies are resolved before they are needed.
  5. Optimise space: discard states no longer needed.

The running time is (number of states) $\times$ (time per state), assuming each state is computed once.
