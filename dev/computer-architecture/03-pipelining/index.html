3. Pipelining

Rather than fully completing one instruction before starting the next, a pipelined processor overlaps different instructions' fetch-decode-execute stages (Chapter 2) simultaneously — this chapter covers how pipelining improves throughput, and the hazards that make it more complicated than it first appears.

The Basic Pipelining Idea

Definition — Instruction Pipelining

Pipelining divides instruction execution into several distinct stages (e.g., fetch, decode, execute, memory access, write-back) and processes different instructions at different stages simultaneously — while one instruction is being decoded, the next is already being fetched, and the one before is already executing — dramatically improving instruction throughput even though any single instruction still takes the same total number of cycles to fully complete.

Structural Hazards

A structural hazard occurs when two instructions in different pipeline stages simultaneously need the same hardware resource (e.g., both needing to access memory at once) — resolved either by stalling one instruction briefly or by duplicating the contended resource so both can proceed.

Data Hazards

Definition — Data Hazard

A data hazard occurs when an instruction depends on the result of a preceding instruction that has not yet completed and written back its result — naively, the dependent instruction would read a stale value. Forwarding (bypassing) resolves most such hazards by routing a result directly from where it was computed to where it's needed, without waiting for it to be formally written back to a register first; when forwarding alone cannot resolve a hazard (e.g., a value not yet available at all), the pipeline must instead stall.

Control Hazards

Definition — Control Hazard

A control hazard arises from branch instructions — the pipeline does not know which instruction to fetch next until the branch itself is resolved, several stages later. Branch prediction (Chapter 6) guesses the likely outcome and speculatively continues fetching down that path, paying a real penalty (flushing incorrectly fetched instructions) only when the prediction turns out wrong.

Pipeline Depth and Diminishing Returns

Remark — Why Pipelines Don't Grow Indefinitely

A deeper pipeline (more, shorter stages) can in principle run at a higher clock frequency, but also increases the penalty paid for any hazard requiring a stall or flush, and demands progressively more sophisticated hazard-avoidance logic (including branch prediction, Chapter 6) — a tradeoff that has led modern processor designs to settle on moderate pipeline depths rather than pursuing ever-deeper pipelines indefinitely.
