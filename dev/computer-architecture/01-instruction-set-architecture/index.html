1. Instruction Set Architecture

The instruction set architecture (ISA) is the contract between hardware and software — the complete vocabulary of instructions a processor understands, and the interface every compiler (Compilers) ultimately targets.

What an ISA Defines

Definition — Instruction Set Architecture

An ISA specifies the set of instructions a processor can execute, the registers available to hold values, the memory addressing modes supported, and the data types operations can act on — defining precisely what a processor must be capable of, while leaving how it actually implements that capability (Chapter 2) entirely up to the specific hardware design.

CISC versus RISC

Two Design Philosophies
  • CISC (Complex Instruction Set Computer) — provides a large set of instructions, some quite complex (a single instruction might perform a memory load, an arithmetic operation, and a store), reducing the number of instructions needed per program at the cost of more complex, harder-to-pipeline (Chapter 3) hardware. The x86 architecture is the most prominent surviving CISC example.
  • RISC (Reduced Instruction Set Computer) — provides a smaller set of simple, uniform instructions (typically one cycle each in the ideal case), trading a higher instruction count per program for hardware that is simpler to pipeline and optimize. ARM and RISC-V are widely used RISC architectures today.

Registers and Addressing Modes

Registers are the small number of extremely fast storage locations built directly into the processor, used to hold values actively being computed on — far faster to access than main memory (Chapter 4), but far more limited in number. An ISA's addressing modes specify the different ways an instruction can identify a memory location to operate on (directly, via a register holding an address, with an offset, and so on).

Instruction Encoding

Every instruction is ultimately encoded as a fixed or variable-length sequence of bits — RISC architectures typically favour fixed-length encoding (simplifying the hardware that decodes instructions, at the cost of some encoding efficiency), while CISC architectures often use variable-length encoding to accommodate their larger, more varied instruction repertoire.

The ISA as a Stable Interface

Remark — Why the ISA Rarely Changes

Because software is compiled against a specific ISA, and binary compatibility across processor generations is enormously valuable commercially, an ISA changes far more slowly and conservatively than the underlying hardware implementation does — successive x86 processor generations have added new instructions and dramatically changed their internal implementation (Chapters 2–6) while remaining able to run software compiled decades earlier.
