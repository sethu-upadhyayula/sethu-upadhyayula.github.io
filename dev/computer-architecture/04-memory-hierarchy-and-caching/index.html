4. Memory Hierarchy and Caching

Main memory is far too slow to keep up with a modern processor's execution speed — this chapter covers the memory hierarchy, a series of progressively smaller and faster memory levels that together bridge this gap.

The Memory Hierarchy

Definition — Memory Hierarchy

A processor's memory hierarchy spans registers (Chapter 1, fastest and smallest), one or more levels of cache (L1, L2, sometimes L3 — each level progressively larger but slower than the one before), main memory (DRAM), and secondary storage (Chapter 8 of Operating Systems, slowest and largest) — designed around the observation that programs exhibit strong locality of reference, so caching the small fraction of data actively being used captures most of the benefit of fast memory without needing to make the entire memory pool that fast (and expensive).

Locality of Reference

Temporal and Spatial Locality
  • Temporal locality — recently accessed data is likely to be accessed again soon (e.g., a loop variable).
  • Spatial locality — data near a recently accessed address is likely to be accessed soon too (e.g., sequential array elements) — which is why caches load an entire "block" or "line" of nearby memory on a miss, not just the single requested byte.

Cache Organization

A cache is organized into fixed-size lines (or blocks), each holding a contiguous chunk of memory and tagged with the address it currently represents. A direct-mapped cache allows each memory address to map to exactly one possible cache line (simple, but prone to unnecessary conflicts); a fully associative cache allows any line to hold any address (flexible, but expensive to search); and a practical set-associative cache is the common compromise between these two extremes.

Cache Hits, Misses, and Replacement

A cache hit finds the requested data already present; a cache miss requires fetching it from the next, slower level of the hierarchy — and, if the cache is full, evicting an existing line, using replacement policies conceptually similar to the page replacement algorithms covered in Chapter 6 of Operating Systems, though implemented directly in fast hardware rather than OS software given the much tighter timing constraints involved.

Write Policies

Remark — Write-Through versus Write-Back

A write-through cache immediately propagates every write to the next memory level, keeping them always in sync at some performance cost; a write-back cache instead marks a modified line "dirty" and defers writing it back until the line is evicted, reducing memory traffic for data written repeatedly before eventually needing to persist — at the cost of added complexity, particularly for the cache coherence problem covered in Chapter 5.
