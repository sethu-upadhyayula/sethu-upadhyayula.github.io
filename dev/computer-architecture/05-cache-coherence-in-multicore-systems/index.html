5. Cache Coherence in Multicore Systems

When multiple processor cores each maintain their own private cache (Chapter 4) but share the same underlying main memory, a new problem emerges: what happens when two cores cache the same memory location, and one of them modifies it? This chapter covers cache coherence, the mechanism ensuring all cores still see a consistent view of memory.

The Coherence Problem

Example — Stale Cached Data Across Cores

If core A and core B both cache the same memory line, and core A modifies its cached copy, core B's cached copy becomes stale unless something actively updates or invalidates it — without a coherence mechanism, core B could continue reading an outdated value indefinitely, silently breaking any program relying on shared memory (Chapter 3 of Operating Systems) to coordinate correctly between threads running on different cores.

Coherence Protocols

A cache coherence protocol defines rules that keep multiple cores' cached copies of the same memory consistent — typically via either an invalidation-based approach (a core writing to a shared line invalidates all other cores' copies, forcing them to re-fetch the updated value on next access) or a (less common) update-based approach (broadcasting the new value directly to other cached copies).

MESI and Related Protocols

Definition — The MESI Protocol

The widely used MESI protocol tags each cache line with one of four states — Modified (dirty, exclusively held by this cache), Exclusive (clean, exclusively held), Shared (clean, potentially held by multiple caches), and Invalid — with cores communicating state transitions over a shared interconnect (a bus, or a more scalable directory-based structure on larger multicore systems) to keep every core's view of shared memory consistent.

Coherence versus Consistency

Remark — A Related but Distinct Concept

Cache coherence (this chapter) ensures all cores eventually see the same value for a given memory location; memory consistency models address the separate, subtler question of what order different cores are allowed to observe a sequence of memory operations from multiple locations occurring in — a distinction that becomes especially important for correctly implementing the low-level synchronization primitives underlying the higher-level locks and semaphores covered in Chapter 3 of Operating Systems.

Performance Implications: False Sharing

Because coherence operates at the granularity of an entire cache line, two cores modifying two entirely unrelated variables that happen to reside on the same cache line can trigger unnecessary coherence traffic between them — a performance pitfall called false sharing, addressed in practice by deliberately padding or aligning frequently-modified shared data structures so unrelated variables don't share a line at all.
