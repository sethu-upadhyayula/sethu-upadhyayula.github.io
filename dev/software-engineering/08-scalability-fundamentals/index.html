8. Scalability Fundamentals

This closing chapter introduces the practical building blocks engineers reach for when a single machine, or a single database, can no longer handle a system's load — a practical, applied counterpart to the more theoretical treatment of distributed coordination covered in Distributed Systems.

Vertical versus Horizontal Scaling

Definition — Scaling Up versus Scaling Out

Vertical scaling ("scaling up") adds more resources (CPU, memory) to a single existing machine — simple, requiring no architectural change, but ultimately limited by the largest machine available and offering no redundancy if that one machine fails. Horizontal scaling ("scaling out") adds more machines, distributing load across them — offering both greater ultimate scale and redundancy, at the cost of the added architectural complexity covered throughout this chapter.

Load Balancing

A load balancer distributes incoming requests across multiple backend server instances, using a strategy such as round-robin, least-connections, or hashing on some request property — allowing horizontal scaling of stateless application servers, and (by routing around servers detected as unhealthy) also contributing to overall system availability.

Caching

Definition — Application-Level Caching

An application-level cache (such as Redis or Memcached) stores frequently accessed or expensive-to-compute data in fast memory, avoiding repeated, costly trips to a database (Database Systems) or other backing service — conceptually the same locality-of-reference idea underlying CPU caching (Chapter 4 of Computer Architecture), just applied at the scale of an entire distributed application rather than within a single processor.

Database Scaling: Replication and Sharding

Two Approaches to Scaling a Database
  • Read replication — maintains one or more read-only copies of a database, directing read traffic to replicas and write traffic to a single primary — effective when a workload is read-heavy, though it introduces the replication-lag consistency questions covered in Distributed Systems.
  • Sharding (partitioning) — splits a dataset across multiple independent database instances (commonly by a hash or range of some key), allowing both read and write load to scale horizontally, at the cost of making any query spanning multiple shards considerably more complex than it would be against a single, unsharded database.

Statelessness and Scalability

Remark — Why Statelessness Simplifies Horizontal Scaling

Application servers designed to be stateless — storing no request-specific data locally between requests, with any needed state kept in a shared database or cache instead — can be scaled horizontally very simply, since a load balancer can route any request to any server instance interchangeably. Servers that instead hold meaningful local state between requests are considerably harder to scale this way, since a client's subsequent requests may need to reach that exact same server instance again.

Where This Leads

These techniques address the practical mechanics of scaling a system's throughput and availability; the deeper theoretical questions they raise — how to keep data consistent across replicas and shards, how to reach agreement across many machines, and how to tolerate individual machine failures gracefully — are covered in full in Distributed Systems.
