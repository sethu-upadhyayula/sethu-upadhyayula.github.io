6. Testing

Automated tests provide confidence that software behaves correctly, both today and after every future change — this chapter covers the standard levels of testing and how test-driven development uses tests as a design tool, not merely a correctness check.

The Testing Pyramid

Definition — The Testing Pyramid

The testing pyramid describes a recommended proportion of tests at each level: many fast, isolated unit tests (testing a single function or class in isolation) at the base, fewer integration tests (testing how multiple components work together, e.g., a service and its database) in the middle, and fewer still, slower end-to-end tests (testing an entire user-facing flow through the real, fully assembled system) at the top — reflecting that unit tests are cheap and fast to run but verify less at once, while end-to-end tests verify more but are slow, brittle, and expensive to maintain.

Test Doubles

Mocks, Stubs, and Fakes
  • Stub — returns predetermined, canned responses, without any real logic behind them.
  • Mock — additionally verifies that specific interactions with it actually occurred as expected during the test.
  • Fake — a simplified but genuinely working implementation (e.g., an in-memory database standing in for a real one) — more realistic than a stub or mock, at the cost of more effort to build and maintain.

Dependency injection (Chapter 3) is what makes substituting a test double for a real dependency practical — a class depending on an injected interface can be tested with a test double implementing that same interface, without needing to modify the class under test at all.

Test-Driven Development

Definition — Test-Driven Development (TDD)

Test-driven development writes a failing test for a small piece of desired behaviour before writing the implementation code that satisfies it, then writes just enough code to make the test pass, then refactors — repeating this red-green-refactor cycle continuously. Proponents argue this produces more testable, better-factored code as a natural side effect of writing the test first, rather than treating testing as an afterthought bolted on once implementation is already "done."

Code Coverage

Code coverage measures what proportion of a codebase's lines (or branches) are executed by the test suite — a useful signal for finding completely untested code, but a poor proxy for actual test quality on its own, since a test can execute a line of code without meaningfully verifying its correctness at all.

Testing in Continuous Integration

Automated tests are most valuable when run automatically and consistently — typically as part of a continuous integration pipeline (Chapter 7) that runs the full test suite against every proposed code change, catching regressions before they reach production rather than relying on manual testing discipline alone.
