7. Version Control and CI/CD

This chapter covers the tooling that makes modern collaborative software development practical at all — version control for tracking and merging code changes, and CI/CD pipelines for automating testing and deployment.

Version Control with Git

Definition — Git

Git is a distributed version control system, tracking a project's complete history as a graph of commits, each capturing a snapshot of the project at a point in time along with a reference to its parent commit(s) — "distributed" meaning every developer's local repository holds the complete history, not just a working copy checked out from a single central server, allowing most operations (viewing history, committing, branching) to happen entirely offline.

Branching Strategies

A branch is an independent line of development, allowing work on a new feature or fix to proceed without disturbing the stable main line of code — teams adopt varying branching strategies (from simple, short-lived feature branches merged frequently, to more elaborate schemes with separate long-lived development and release branches) trading off integration frequency against isolation of in-progress work.

Code Review and Pull Requests

A pull request (or merge request) proposes merging a branch's changes into another (typically the main line), providing a natural point for code review — other team members inspecting proposed changes before they're merged, catching bugs, design issues, or maintainability concerns (Chapter 2) earlier and more cheaply than they would be caught after reaching production.

Continuous Integration

Definition — Continuous Integration (CI)

Continuous integration automatically builds and runs the test suite (Chapter 6) against every proposed code change, ideally before it's even merged — catching integration problems and regressions quickly and consistently, rather than allowing many developers' changes to accumulate and only be tested together in one large, hard-to-debug batch much later.

Continuous Deployment

Definition — Continuous Deployment (CD)

Continuous deployment extends CI by automatically deploying every change that passes the pipeline directly to production (or, in the more conservative continuous delivery variant, keeping every passing change ready to deploy with a manual approval step) — enabling the frequent, small, low-risk releases favoured by the DevOps philosophy introduced in Chapter 1, as opposed to large, infrequent, high-risk release events.

Deployment Strategies

Remark — Reducing Deployment Risk

Techniques such as blue-green deployment (running old and new versions side by side, switching traffic over once the new version is verified healthy) and canary releases (rolling a change out to a small fraction of traffic first, expanding only if no problems appear) reduce the risk of any single deployment causing a widespread outage — a natural complement to the rapid release cadence CI/CD enables, since frequent releases are only genuinely safe if each individual release's blast radius can be kept small.
