5. API Design

An API is a contract — this chapter covers how to design one that is clear to use correctly, hard to use incorrectly, and stable enough that consumers aren't constantly broken by changes on the other side of it.

REST

Definition — REST

REST (Representational State Transfer) is an architectural style for web APIs built around HTTP (Chapter 6 of Computer Networks) — resources are identified by URLs, and standard HTTP methods (GET, POST, PUT, DELETE) express the operation to perform on that resource, with responses typically encoded as JSON. RESTful design favours resource-oriented thinking (nouns, like "orders" or "users") over action-oriented endpoints (verbs, like "createOrder").

Versioning

Because an API's consumers cannot all update simultaneously with the API itself, breaking changes require a deliberate versioning strategy — commonly a version number in the URL path or a request header — allowing existing consumers to keep using an older version while new consumers adopt the latest, and giving the API owner a clear, bounded process for eventually deprecating and retiring old versions.

Idempotency

Definition — Idempotency

An operation is idempotent if performing it multiple times has the same effect as performing it once — GET, PUT, and DELETE are conventionally expected to be idempotent under REST, while POST typically is not. Idempotency matters enormously for reliability: a client that doesn't receive a response (due to a network issue, Computer Networks) can safely retry an idempotent request without risking an unintended duplicate effect (such as double-charging a customer).

Pagination and Rate Limiting

APIs returning potentially large result sets typically implement pagination (returning results in bounded-size pages, with a token or offset to fetch the next page), and APIs exposed to external or high-volume consumers typically implement rate limiting (capping how many requests a given client can make in a given time window) — both protecting the API's own backing systems from being overwhelmed by any single consumer's usage pattern.

Beyond REST: GraphQL and gRPC

Alternative API Styles
  • GraphQL — lets a client specify exactly which fields it needs in a single request, avoiding both the over-fetching and the multiple-round-trip under-fetching that a fixed REST endpoint shape can cause.
  • gRPC — a high-performance RPC framework using a strongly typed schema (Protocol Buffers) and HTTP/2, common for efficient internal service-to-service communication (particularly relevant to microservices, Chapter 4) rather than public-facing APIs.

Documentation as Part of the API

An API is only as usable as its documentation — clear, accurate, example-rich documentation (often generated directly from a formal schema, such as an OpenAPI specification) is not a secondary concern but a core part of what makes an API actually usable by other developers, arguably as important as the API's own technical correctness.
