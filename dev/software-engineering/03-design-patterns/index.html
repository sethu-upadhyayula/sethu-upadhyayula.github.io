3. Design Patterns

Design patterns are named, reusable solutions to design problems that recur across many different codebases — a shared vocabulary that lets engineers communicate a design idea precisely without re-explaining it from scratch every time.

Creational Patterns

Patterns for Object Creation
  • Factory — delegates object creation to a dedicated method or class, decoupling client code from the specific concrete classes being instantiated.
  • Singleton — ensures a class has exactly one instance, globally accessible — useful in limited cases, but often criticized for introducing hidden global state that complicates testing and reasoning about a system.
  • Builder — separates the construction of a complex object from its representation, useful when an object has many optional configuration parameters that would otherwise require an unwieldy constructor.

Structural Patterns

Patterns for Composing Objects
  • Adapter — converts one interface into another that client code expects, letting otherwise-incompatible components work together.
  • Decorator — attaches additional responsibilities to an object dynamically, as an alternative to creating many specialized subclasses for every possible combination of behaviour.
  • Facade — provides a simplified, unified interface to a more complex subsystem of classes.

Behavioural Patterns

Patterns for Object Interaction
  • Observer — one object (the subject) notifies a list of dependent objects (observers) automatically of any state change, the basis for many event-handling and reactive systems.
  • Strategy — encapsulates an interchangeable family of algorithms behind a common interface, letting the specific algorithm used vary independently of the code that uses it.
  • Command — encapsulates a request as an object, enabling features like undo/redo, queuing, and logging of operations that would be awkward to implement if a request were just a direct method call.

Dependency Injection

Definition — Dependency Injection

Dependency injection supplies an object's dependencies from outside rather than having the object construct them itself internally — the standard practical technique for achieving the Dependency Inversion principle (Chapter 2), and particularly valuable for testability (Chapter 6), since a dependency can be swapped for a test double without modifying the class that depends on it at all.

Patterns as a Means, Not an End

Remark — Avoiding Pattern Overuse

Design patterns solve specific, recurring problems — applying one where the underlying problem doesn't actually exist adds unnecessary indirection and complexity for no real benefit, directly conflicting with the KISS and YAGNI principles covered in Chapter 2. Recognizing when a pattern genuinely fits a problem, rather than reaching for patterns reflexively, is itself a significant part of the skill design patterns are meant to support.
