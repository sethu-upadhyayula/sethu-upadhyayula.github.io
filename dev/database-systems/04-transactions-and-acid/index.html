4. Transactions and ACID

Real applications frequently need to perform several related database operations as a single logical unit - transferring money between two accounts requires debiting one and crediting the other, and either both must happen or neither should. This chapter covers transactions, the mechanism providing this guarantee.

What a Transaction Is

Definition - Transaction

A transaction is a sequence of one or more database operations treated as a single, indivisible logical unit - either fully committed (all its effects become permanent) or fully rolled back (none of its effects persist), with no possibility of a partial, inconsistent result becoming visible to other parts of the system.

The ACID Properties

Definition - ACID

Atomicity - a transaction's operations either all happen or none do.
Consistency - a transaction moves the database from one valid state (satisfying all declared constraints, Chapter 2) to another.
Isolation - concurrently running transactions do not observe each other's intermediate, uncommitted state (Chapter 5 covers the practical nuances here in depth).
Durability - once a transaction commits, its effects survive any subsequent crash or power failure.

Achieving Durability: Write-Ahead Logging

Durability is typically implemented via write-ahead logging (WAL) - before modifying the actual data on disk, the database first writes a durable log record describing the intended change, so that after a crash, the log can be replayed to redo any committed-but-not-yet-fully-applied changes, or undo any uncommitted ones - directly analogous to the journaling technique covered in Chapter 7 of Operating Systems.

Achieving Atomicity: Commit and Rollback

A transaction ends either with COMMIT (making all its changes permanent and visible) or ROLLBACK (discarding all its changes as though they never happened) - the database ensures atomicity by only making a transaction's effects durable and visible at the single instant of commit, never partway through.

Savepoints

Remark - Partial Rollback with Savepoints

A savepoint marks an intermediate point within a longer transaction, allowing a rollback to that specific point rather than discarding the entire transaction - useful when a transaction includes an operation that might fail in a way the application wants to handle and recover from without abandoning everything done in that transaction so far.
