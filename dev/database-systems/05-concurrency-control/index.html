5. Concurrency Control

Databases must support many transactions running concurrently for acceptable performance, but naive concurrent execution can violate the isolation property introduced in Chapter 4 - this chapter covers how databases actually make concurrent transactions behave correctly.

Serializability

Definition - Serializability

A concurrent execution of transactions is serializable if its final result is equivalent to some sequential (one-at-a-time) execution of those same transactions - the gold-standard correctness criterion for isolation, since it guarantees concurrency never produces a result that couldn't have arisen from running transactions one after another in some order.

Concurrency Anomalies

Isolation Problems Without Proper Control
  • Dirty read - reading data written by an uncommitted transaction that might still be rolled back.
  • Non-repeatable read - reading the same row twice within one transaction and getting different values, because another transaction committed a change in between.
  • Phantom read - re-running the same query within one transaction and getting a different set of rows, because another transaction inserted or deleted qualifying rows in between.

Locking

Definition - Two-Phase Locking

Two-phase locking (2PL) requires a transaction to acquire all the locks it needs (shared locks for reads, exclusive locks for writes) before releasing any of them - a "growing" phase of acquiring locks strictly followed by a "shrinking" phase of releasing them - a sufficient (though not the only possible) condition for guaranteeing serializability, at the cost of the deadlock risk covered in Chapter 4 of Operating Systems whenever transactions acquire locks in inconsistent orders.

Isolation Levels

Remark - Trading Isolation for Performance

Full serializability can be expensive to enforce, so SQL defines weaker isolation levels - read uncommitted, read committed, repeatable read, and serializable - each preventing a different subset of the anomalies above, letting an application deliberately trade some isolation guarantee for improved concurrency and throughput when its specific correctness needs allow it.

Multiversion Concurrency Control

Many modern databases use multiversion concurrency control (MVCC) instead of relying purely on locking - maintaining multiple versions of a row simultaneously, so a read can be served from an appropriately old version without blocking a concurrent writer at all, substantially improving read concurrency compared to lock-based approaches that would otherwise force readers to wait on writers.
