2. SQL: Querying Relational Data

SQL (Structured Query Language) is the standard language for interacting with relational databases (Chapter 1) - this chapter covers its core constructs, from basic queries through the joins and aggregations that make relational algebra practically usable.

DDL, DML, and DQL

SQL's Sub-Languages
  • DDL (Data Definition Language) - CREATE, ALTER, DROP: defining and modifying schema.
  • DML (Data Manipulation Language) - INSERT, UPDATE, DELETE: modifying data.
  • DQL (Data Query Language) - SELECT: retrieving data, the most frequently used and most elaborated part of SQL in practice.

The SELECT Statement

A basic SELECT statement specifies which columns to retrieve, which table(s) to retrieve them from, and a WHERE clause filtering which rows qualify - directly corresponding to the projection and selection operations of relational algebra (Chapter 1), with GROUP BY and aggregate functions (COUNT, SUM, AVG, and others) supporting summarization across groups of rows.

Joins

Types of Join
  • Inner join - returns only rows with matching values in both tables.
  • Left (outer) join - returns all rows from the left table, with unmatched right-table columns filled as NULL.
  • Right and full outer joins - analogous, preserving unmatched rows from the right table, or from both tables respectively.

Joins are the relational model's primary mechanism for combining data spread across multiple, normalized tables (Chapter 3) back into a single result set - how a database engine actually executes a join efficiently is covered in Chapters 6–7.

Subqueries and Common Table Expressions

A subquery nests one query inside another, often within a WHERE or FROM clause, while a common table expression (CTE) (a WITH clause) names a subquery's result for reuse elsewhere in the same statement - both allow complex queries to be built up from simpler logical pieces, sometimes with performance implications depending on how the query optimizer (Chapter 7) chooses to execute them.

Constraints

SQL schema definitions can declare constraints - NOT NULL, UNIQUE, PRIMARY KEY, FOREIGN KEY (Chapter 1), and CHECK - that the database engine enforces automatically on every insert or update, rejecting any change that would violate them, providing a first line of data-integrity defense before application code even gets involved.
