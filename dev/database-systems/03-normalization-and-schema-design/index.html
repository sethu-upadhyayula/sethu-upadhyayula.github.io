3. Normalization and Schema Design

How data is split across tables significantly affects a database's integrity and maintainability - this chapter covers normalization, the formal process of structuring a schema to eliminate redundancy and the data anomalies it causes.

The Problem: Update Anomalies

Example - Anomalies from a Poorly Normalized Table

A single table storing both order details and repeated customer address information for every order risks an update anomaly (updating a customer's address requires changing every one of their order rows, and missing even one leaves inconsistent data), an insertion anomaly (a new customer with no orders yet cannot be recorded at all), and a deletion anomaly (deleting a customer's only order accidentally erases their address information as a side effect).

Functional Dependencies

Definition - Functional Dependency

A functional dependency $A \to B$ holds if knowing the value of attribute(s) $A$ always determines the value of attribute(s) $B$ within a relation - the formal concept normalization uses to identify and eliminate redundancy, since a functional dependency not fully rooted in a table's primary key is generally a sign that data ought to be split into a separate table.

Normal Forms

The First Three Normal Forms
  • 1NF - every attribute holds a single, atomic value (no repeating groups or multi-valued columns).
  • 2NF - 1NF, plus every non-key attribute depends on the entire primary key, not just part of it (relevant only for composite keys).
  • 3NF - 2NF, plus no non-key attribute depends on another non-key attribute (eliminating "transitive" dependencies) - the practical target for most real-world schema design.

Beyond 3NF

Stricter normal forms - BCNF (Boyce-Codd Normal Form), 4NF, and 5NF - address progressively subtler forms of redundancy, but are less commonly pursued explicitly in everyday schema design, since 3NF already eliminates the great majority of practically significant anomalies for most applications.

Denormalization

Remark - When Redundancy Is Reintroduced Deliberately

Denormalization deliberately reintroduces some redundancy - duplicating a piece of data across tables, or precomputing an aggregate - to improve read performance, at the cost of reintroducing some of the update-anomaly risk normalization was designed to eliminate. This tradeoff is made deliberately and selectively, typically only after identifying a specific, measured performance bottleneck that normalized joins (Chapter 2) are causing, rather than as a default starting design choice.
