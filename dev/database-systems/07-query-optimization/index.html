7. Query Optimization

A single SQL query typically has many possible execution strategies that all produce the identical, correct result - this chapter covers how the query optimizer chooses among them, aiming for the fastest one without requiring the application developer to specify execution details manually.

Logical versus Physical Plans

A query is first translated into a logical plan - a tree of relational algebra operations (Chapter 1) representing what the query means - which the optimizer then transforms (reordering joins, pushing filter conditions as early as possible) and ultimately converts into a physical plan, specifying exactly which algorithm and index (Chapter 6) to use for each operation.

Join Algorithms

Common Physical Join Algorithms
  • Nested loop join - for each row of one table, scan the other table for matches; simple, but potentially very slow for large tables without an index to speed the inner scan.
  • Hash join - builds an in-memory hash table on the smaller input, then probes it with the larger input; efficient for large, unindexed equi-joins.
  • Sort-merge join - sorts both inputs on the join key, then merges them in a single linear pass; efficient when inputs are already sorted (e.g., via a clustered index, Chapter 6) or when the join result itself needs to be sorted anyway.

Cost-Based Optimization

Definition - Cost-Based Optimizer

A cost-based optimizer estimates the resource cost (typically disk I/O and CPU time) of many candidate physical plans, using statistics maintained about table sizes and the distribution of values in indexed columns, and selects the plan with the lowest estimated cost - rather than exhaustively executing every candidate plan to see which is actually fastest, which would defeat the purpose of optimization.

Join Order

For a query joining several tables, the order in which those joins are performed can dramatically affect performance - joining two small, highly selective tables first, before bringing in a much larger table, generally produces a far smaller intermediate result than joining the large table in first - so choosing a good join order is one of the optimizer's most performance-critical decisions, and one that grows combinatorially harder as more tables are involved.

Reading Execution Plans in Practice

Remark - EXPLAIN and Practical Tuning

Most SQL databases provide an EXPLAIN (or similar) command that reveals the actual execution plan chosen for a given query - an essential practical tool for diagnosing why a specific query performs poorly, often revealing that an expected index (Chapter 6) isn't being used, or that outdated table statistics are causing the optimizer to badly misestimate a plan's true cost.
