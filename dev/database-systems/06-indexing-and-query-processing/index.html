6. Indexing and Query Processing

Without additional structure, answering even a simple query would require scanning every row in a table - this chapter covers indexing, the mechanism that lets a database find matching rows dramatically faster.

What an Index Is

Definition - Index

An index is an auxiliary data structure, built on one or more columns of a table, that allows the database to quickly locate rows matching a given value or range without scanning the entire table - at the cost of extra storage space and the overhead of maintaining the index whenever the underlying table is modified.

B-Tree Indexes

Definition - B-Tree

The B-tree (and its common variant, the B+-tree) is the standard index structure in most relational databases - a balanced, wide-fanout tree structure that keeps lookup, insertion, and deletion operations efficient (logarithmic in the number of rows) even for enormous tables, and is specifically designed around the block-based nature of disk I/O (Chapter 8 of Operating Systems), minimizing the number of disk block reads needed per operation.

Hash Indexes

A hash index supports very fast exact-match lookups by hashing the indexed value to a bucket location, but - unlike a B-tree - cannot efficiently support range queries (finding all rows with a value between two bounds), since a hash function deliberately scatters similar values across unrelated buckets rather than preserving their relative order.

Clustered versus Non-Clustered Indexes

Definition - Clustered Index

A clustered index determines the actual physical storage order of a table's rows on disk (a table can have at most one), while a non-clustered (secondary) index is a separate structure pointing back to the row's location, allowing multiple non-clustered indexes on different columns of the same table.

Query Execution Plans

Given a query, the database's query processor chooses a specific execution plan - which indexes to use (if any), which join algorithm to apply for each join (Chapter 2), and in what order to combine multiple tables - a choice covered in depth in Chapter 7, since a poorly chosen plan can be orders of magnitude slower than a well-chosen one for the exact same logical query.
