7. Partitioning and Consistent Hashing

Chapter 8 of Software Engineering & Systems Design introduced database sharding at a practical level — this chapter goes deeper into how a dataset is actually assigned across nodes, and specifically how that assignment can adapt smoothly as nodes are added or removed.

Partitioning Strategies

Range versus Hash Partitioning
  • Range partitioning — assigns contiguous ranges of a key's value to each partition; supports efficient range queries, but risks uneven load if keys aren't uniformly distributed (a common real-world case: sequentially increasing timestamp-based keys concentrate all recent writes on a single "hot" partition).
  • Hash partitioning — applies a hash function to a key to determine its partition, distributing load much more evenly regardless of the underlying key distribution, at the cost of losing the ability to efficiently answer range queries across keys, since a hash function deliberately scatters similar keys unpredictably.

The Problem with Naive Hash Partitioning

A naive approach — computing hash(key) mod N for $N$ partitions — works fine while $N$ stays fixed, but changing $N$ (adding or removing a node) remaps almost every key to a different partition, forcing a massive, disruptive data reshuffle across the entire cluster just to accommodate one node joining or leaving.

Consistent Hashing

Consistent Hashing

Consistent hashing maps both nodes and keys onto a single, shared hash space (conceptually, a circle), with each key assigned to the first node reached moving clockwise from that key's own hash position. Adding or removing a single node then only reassigns the (small, bounded) portion of key space between that node and its clockwise neighbour, rather than reshuffling the entire dataset — dramatically reducing the data movement cost of scaling a partitioned system up or down compared to naive modulo-based hashing.

Virtual Nodes

Remark — Improving Load Balance

Plain consistent hashing can still produce uneven load if nodes happen to land unevenly around the hash circle — virtual nodes address this by having each physical node occupy many positions around the circle rather than just one, smoothing out load distribution across physical nodes considerably, at the cost of somewhat more bookkeeping to track which virtual positions belong to which physical node.

Rebalancing in Practice

Even with consistent hashing minimizing the amount of data that needs to move, actually executing a rebalance — safely transferring data between nodes while the system continues serving live traffic — remains an operationally significant concern in any large partitioned system, connecting the elegant hashing theory covered in this chapter to the practical replication and consistency mechanisms covered in Chapter 4 that must remain correct throughout the rebalancing process itself.
