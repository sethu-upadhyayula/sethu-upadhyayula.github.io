1. Models of Distributed Computation

Before reasoning about any distributed algorithm, it's necessary to be precise about what a system can actually assume regarding timing and failure — this chapter covers the standard models used to make those assumptions explicit.

Why Distributed Systems Are Hard

Remark — The Core Difficulty

A single machine either works or it doesn't, and its own internal operations happen in a well-defined order. A distributed system has neither guarantee — any individual machine can fail independently of the others, and the network (Computer Networks) connecting them can delay, reorder, drop, or duplicate messages — meaning a distributed algorithm must remain correct despite these failures and irregularities, not merely in the easy case where everything works as expected.

Synchronous versus Asynchronous Models

Definition — Synchronous and Asynchronous Systems

A synchronous system model assumes known upper bounds on message delivery time and processing speed, making it possible to reliably distinguish a slow machine from a failed one (it simply hasn't responded within the known bound). An asynchronous system model makes no such timing assumptions at all — a message might take arbitrarily long to arrive, making it fundamentally impossible to distinguish a slow process from a failed one with certainty. Most real-world distributed systems are best modeled as asynchronous (or "partially synchronous," assuming bounds hold eventually but not always), since real networks provide no hard timing guarantees.

Failure Models

Types of Failure a System Must Tolerate
  • Crash failure — a process simply stops, and never sends any further (correct or incorrect) messages.
  • Omission failure — a process fails to send or receive some messages, but otherwise behaves correctly.
  • Byzantine failure — a process behaves arbitrarily, potentially sending incorrect or even maliciously crafted messages — the strongest, most general failure model, and correspondingly the hardest to tolerate.

The FLP Impossibility Result

The FLP Result

The FLP impossibility result (Fischer, Lynch, and Paterson) proves that in a fully asynchronous system, no consensus algorithm (Chapter 3) can guarantee both correctness and termination if even a single process might fail — a foundational, sobering theoretical limit that every practical consensus algorithm must work around, typically by making some additional timing assumption (even a weak, eventually-true one) rather than remaining purely, permanently asynchronous.

Why These Models Matter in Practice

Understanding which model a given system assumes explains both what guarantees it can actually make and why — a system designed under a synchronous assumption that is deployed on a real, asynchronous network can fail in surprising ways precisely because its correctness proof relied on an assumption the real network doesn't actually satisfy, a recurring theme throughout the rest of this subject.
