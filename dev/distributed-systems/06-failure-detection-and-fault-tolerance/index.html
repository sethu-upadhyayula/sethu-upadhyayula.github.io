6. Failure Detection and Fault Tolerance

As Chapter 1 noted, the asynchronous model makes it impossible to perfectly distinguish a slow process from a failed one — yet a working system must still make practical decisions about when to treat another machine as failed, and continue operating regardless. This chapter covers how.

Failure Detectors

Definition — Failure Detector

A failure detector is a (necessarily imperfect) mechanism providing information about which processes are suspected to have failed — typically implemented via periodic heartbeat messages, with a process suspected as failed if its heartbeat hasn't been received within some timeout. Because network delay (Chapter 1) can be mistaken for failure, every practical failure detector risks both false positives (wrongly suspecting a live process) and, if timeouts are set too generously, slow detection of genuine failures.

Heartbeats and Timeout Tuning

Choosing a heartbeat interval and timeout involves a direct tradeoff: shorter timeouts detect real failures faster but risk more false positives (and the wasted work of, for instance, triggering an unnecessary leader re-election, Chapter 3); longer timeouts reduce false positives but leave the system briefly using a resource (or trusting a leader) that has, in reality, already failed.

Replication as Fault Tolerance

The replication mechanisms covered in Chapter 4 serve fault tolerance directly — if data (or a service) is replicated across multiple machines, the failure of any single one need not mean the data or service becomes unavailable, provided a sufficient quorum of replicas remain reachable, connecting this chapter's failure-detection concerns directly back to Chapter 4's consistency tradeoffs.

Quorum Systems

Definition — Quorum

A quorum is the minimum number of replicas that must participate in an operation for it to be considered successful — commonly a strict majority, chosen specifically so that any two quorums (e.g., for a write and a subsequent read) are guaranteed to overlap in at least one replica, ensuring a read quorum always includes at least one replica aware of the most recent successfully committed write.

Graceful Degradation

Remark — Designing for Partial Failure

Well-designed distributed systems aim for graceful degradation under partial failure — continuing to serve at least some requests correctly (perhaps with reduced functionality or consistency guarantees, Chapter 4) rather than failing completely the moment any single component becomes unreachable — a design goal that shapes decisions throughout this subject, from how consensus systems are architected (Chapter 3) to how sagas (Chapter 5) handle a failed step rather than abandoning an entire business operation outright.
