2. Time, Clocks, and Event Ordering

Because separate machines' physical clocks are never perfectly synchronized, and network delays (Chapter 1) are variable and unpredictable, a distributed system cannot simply timestamp events with wall-clock time and expect a meaningful, consistent global ordering — this chapter covers the logical alternatives that have been developed instead.

The Problem with Physical Clocks

Even with protocols like NTP actively synchronizing clocks across machines, physical clock drift and network delay mean that two machines' clocks are never exactly aligned — comparing raw timestamps from two different machines to determine "which event happened first" can therefore give an incorrect answer, particularly for events that occurred close together in time.

Lamport's Happens-Before Relation

Definition — Happens-Before

Leslie Lamport's happens-before relation ($\rightarrow$) defines a partial ordering of events based purely on causality rather than physical time: if event $a$ occurs before event $b$ on the same process, $a \rightarrow b$; if $a$ is the sending of a message and $b$ is its receipt, $a \rightarrow b$; and the relation is transitive. Two events neither related by happens-before are concurrent — there is no causal relationship between them, and no meaningful sense in which one "really" happened first.

Lamport Timestamps

Definition — Lamport Timestamp

A Lamport timestamp is a simple counter, incremented before each local event and updated to exceed any timestamp received in a message, such that $a \rightarrow b$ implies the Lamport timestamp of $a$ is less than that of $b$ — providing a cheap, practical way to derive a consistent total ordering of events compatible with the happens-before relation (though the converse doesn't hold: a smaller Lamport timestamp doesn't necessarily mean the event actually happened-before the other, only that no contradiction with happens-before is introduced by ordering them this way).

Vector Clocks

Definition — Vector Clock

A vector clock extends Lamport timestamps to fully capture the happens-before relation (not just an ordering consistent with it) — each process maintains a vector of counters, one per process in the system, allowing any two events' vector clocks to be compared and determine definitively whether one happened-before the other, or whether they are genuinely concurrent — at the cost of a vector whose size grows with the number of processes, compared to a Lamport timestamp's single integer.

Practical Applications

Example — Detecting Conflicting Updates

Systems favouring availability over strict consistency (Chapter 8 of Database Systems's discussion of the CAP theorem) often use vector clocks to detect when two replicas received genuinely concurrent, conflicting updates to the same data — allowing the system to flag the conflict for resolution (either automatically, via a merge rule, or by surfacing it to the application) rather than silently and arbitrarily picking one update to keep.
