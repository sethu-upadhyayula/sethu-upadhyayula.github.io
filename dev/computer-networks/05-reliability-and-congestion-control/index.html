5. Reliability and Congestion Control

Building on Chapter 4's introduction to TCP, this chapter covers the specific mechanisms TCP uses to detect lost data and, just as importantly, to avoid sending faster than the network can actually handle.

Detecting Loss

TCP detects lost data either through a retransmission timeout (no acknowledgment received within an expected time) or through duplicate acknowledgments (the receiver repeatedly acknowledging the same, most-recently-received-in-order byte, signaling that a later segment arrived but an earlier one is still missing) - the latter allowing faster loss detection than waiting for a timeout in many cases.

Flow Control

Definition - Flow Control

Flow control prevents a fast sender from overwhelming a slower receiver's ability to process incoming data, using a receive window the receiver advertises to the sender, indicating how much additional data it is currently willing to buffer - distinct from congestion control (below), which protects the network itself rather than the receiving endpoint specifically.

Congestion Control

Definition - Congestion Control

Congestion control adjusts a TCP sender's transmission rate to avoid overwhelming the network path itself (routers and links between sender and receiver, Chapter 3), inferring congestion primarily from observed packet loss or delay, since individual senders generally cannot directly observe router queue lengths along their path.

Slow Start and Congestion Avoidance

The Classic TCP Congestion Control Algorithm

Classic TCP congestion control begins with slow start, roughly doubling the sending rate (the congestion window) each round trip until a loss event or a threshold is reached, then switches to congestion avoidance, increasing the window more conservatively (roughly linearly) - and, upon detecting loss, sharply reduces the window before resuming this cycle, collectively known as "additive increase, multiplicative decrease" (AIMD).

Modern Congestion Control

Remark - Beyond Classic Loss-Based Control

More recent congestion control algorithms (such as BBR) move away from relying purely on loss as a congestion signal, instead actively modeling the network path's available bandwidth and round-trip time - motivated partly by the observation that on some modern links, meaningful queuing delay (bufferbloat) can build up well before any packet is actually dropped, meaning loss-based algorithms alone react later than they ideally would.
