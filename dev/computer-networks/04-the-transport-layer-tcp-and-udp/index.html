4. The Transport Layer: TCP and UDP

The network layer (Chapter 3) gets a packet from one host to another, but says nothing about which specific application on that host the data is actually for, or whether it arrives reliably - the transport layer's two main protocols, TCP and UDP, address this in two very different ways.

Ports

A port number identifies a specific application or service on a host, allowing a single IP address (Chapter 3) to support many simultaneous network connections to different applications at once - a connection is fully identified by the combination of source IP, source port, destination IP, and destination port.

TCP: Reliable, Ordered Delivery

Definition - TCP

TCP (Transmission Control Protocol) provides a reliable, ordered, connection-oriented byte stream between two endpoints - establishing a connection via a three-way handshake (SYN, SYN-ACK, ACK) before any data is sent, acknowledging received data, retransmitting anything lost, and reordering out-of-order segments before delivering them to the application in the correct sequence.

UDP: Simple, Unreliable Delivery

Definition - UDP

UDP (User Datagram Protocol) provides a much simpler, connectionless service - packets ("datagrams") are sent with minimal overhead and no guarantee of delivery, ordering, or duplicate protection. Applications that need low latency more than perfect reliability (such as real-time voice/video, or DNS, Chapter 6) often choose UDP and implement any additional reliability they specifically need themselves, rather than paying TCP's overhead for guarantees they don't fully require.

Reliability Mechanisms in TCP

TCP achieves reliability through sequence numbers (identifying each byte's position in the stream), acknowledgments (confirming receipt), and retransmission timers (resending data not acknowledged within an expected window) - covered together with TCP's congestion control mechanisms in Chapter 5.

Connection Termination

Remark - Closing a TCP Connection

TCP connections are closed via a four-step handshake (each side independently signals it has finished sending, FIN, and receives acknowledgment) rather than the three-step process used to open a connection - allowing each direction of the full-duplex connection to be closed somewhat independently, since one side may still have data left to send after the other has finished.
