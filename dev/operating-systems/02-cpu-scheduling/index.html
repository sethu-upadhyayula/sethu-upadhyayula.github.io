2. CPU Scheduling

With typically far more ready processes than available CPU cores, the operating system must repeatedly decide which process runs next - the subject of this chapter.

Scheduling Goals

Competing Scheduling Objectives
  • Throughput - maximizing the number of processes completed per unit time.
  • Turnaround time - minimizing the total time from a process's submission to its completion.
  • Waiting time - minimizing time spent in the ready queue rather than actually running.
  • Response time - minimizing the delay before a process starts producing output, particularly important for interactive systems.
  • Fairness - ensuring no process is starved of CPU time indefinitely.

These goals frequently conflict - optimizing purely for throughput can starve short interactive tasks behind long-running ones, while optimizing purely for response time can reduce overall throughput - so real schedulers make deliberate tradeoffs rather than optimizing any single metric in isolation.

Classic Scheduling Algorithms

Common Scheduling Algorithms
  • First-come, first-served (FCFS) - simple, but a long process can make short processes behind it wait unnecessarily (the "convoy effect").
  • Shortest job first (SJF) - provably minimizes average waiting time, but requires knowing (or estimating) a process's burst length in advance.
  • Round robin - each process gets a fixed time slice (quantum) before being preempted back to the ready queue, giving good response time for interactive workloads at some throughput cost from frequent context switching (Chapter 1).
  • Priority scheduling - runs the highest-priority ready process first, requiring an aging mechanism to prevent low-priority process starvation.

Multilevel Feedback Queues

Real-world general-purpose OS schedulers (such as those in Linux and Windows) typically use a multilevel feedback queue - multiple ready queues with different priority levels and time quanta, with a process's queue level adjusted dynamically based on its observed behaviour (an interactive process that frequently blocks on I/O is favoured with higher priority, while a CPU-bound process is progressively demoted) - approximating the practical benefits of several classic algorithms above without requiring advance knowledge of burst lengths.

Multiprocessor Scheduling

Remark - Scheduling Across Multiple Cores

On multi-core systems, scheduling must also consider processor affinity (keeping a process on the same core to preserve cache locality) and load balancing across cores - adding a genuinely distributed-systems-flavoured dimension (Distributed Systems) to what is otherwise a single-machine problem, since cores can be thought of as independent resources that nonetheless share memory and must coordinate.
