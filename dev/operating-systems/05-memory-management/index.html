5. Memory Management

Every process (Chapter 1) needs memory to run in, but physical RAM is a shared, limited resource across all processes on a system - this chapter covers how the OS manages this sharing while giving each process the illusion of its own private memory.

Address Spaces

Definition - Address Space

Each process has its own address space - the range of memory addresses it can use, isolated from every other process's address space by the OS and hardware (the memory management unit, or MMU) - so that one process cannot accidentally (or maliciously) read or corrupt another's memory, and so that every process can be written as though it has the machine's full address range to itself.

Paging

Definition - Paging

Paging divides a process's virtual address space into fixed-size pages and physical memory into equally sized frames, with a per-process page table mapping each virtual page to whichever physical frame currently holds it. Because pages need not be mapped to contiguous physical frames, paging eliminates the external fragmentation problem that plagued older, contiguous-allocation memory management schemes.

The Translation Lookaside Buffer

Because every single memory access would otherwise require consulting the page table (itself stored in memory), hardware includes a translation lookaside buffer (TLB) - a small, fast cache of recent virtual-to-physical page translations - dramatically speeding up the common case where a recently used page's translation is still cached, at the cost of a somewhat more expensive lookup ("TLB miss") when it is not.

Segmentation

Definition - Segmentation

Segmentation divides a process's address space into logically meaningful, variable-sized segments (code, stack, heap, and others) rather than fixed-size pages, more directly matching how a programmer or compiler naturally thinks about a program's structure - though segmentation alone reintroduces external fragmentation, which is why modern systems typically combine it with paging (paging each segment) rather than using pure segmentation on its own.

Multi-Level Page Tables

Remark - Why Page Tables Need Their Own Hierarchy

A single, flat page table for a large modern 64-bit address space would itself require an impractically large amount of memory, most of it mapping unused address regions - multi-level (hierarchical) page tables solve this by making higher-level table entries themselves point to further page tables, allowing large unused regions of the address space to be represented compactly (or not at all), at the cost of requiring multiple sequential memory accesses to fully resolve one address translation before the TLB above caches the result.
