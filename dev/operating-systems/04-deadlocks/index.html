4. Deadlocks

Synchronization primitives (Chapter 3) prevent race conditions, but used incorrectly they can create a different problem entirely - a set of processes permanently stuck waiting on each other, unable to proceed. This chapter covers deadlock: why it happens, and how systems handle it.

The Four Necessary Conditions

The Four Coffman Conditions

A deadlock requires all four of the following to hold simultaneously: mutual exclusion (resources cannot be shared), hold and wait (a process holds one resource while waiting for another), no preemption (a resource cannot be forcibly taken from a process holding it), and circular wait (a cycle of processes, each waiting on a resource held by the next). Breaking any single one of these four conditions is sufficient to prevent deadlock entirely.

Deadlock Prevention

Deadlock prevention designs the system so that at least one of the four conditions can never arise - for example, requiring all processes to request every resource they will ever need at once (eliminating hold-and-wait), or imposing a strict global ordering on resource acquisition (eliminating circular wait, since a cycle is impossible if every process acquires resources in the same fixed order).

Deadlock Avoidance: The Banker's Algorithm

Definition - Banker's Algorithm

Rather than eliminating a condition outright, deadlock avoidance grants resource requests only if doing so leaves the system in a safe state - one from which there exists at least one ordering in which every process can still eventually obtain all its declared maximum resource needs and finish. The Banker's algorithm checks this safety condition before granting each request, refusing (or delaying) any request that would leave the system unsafe, at the cost of requiring processes to declare their maximum resource needs in advance.

Deadlock Detection and Recovery

Rather than preventing or avoiding deadlock proactively, a system can instead allow it to potentially occur and periodically check for it - deadlock detection constructs a resource-allocation graph and checks for cycles (a necessary condition for deadlock when each resource type has a single instance) - and, once detected, recover by forcibly preempting resources from some process or terminating one or more deadlocked processes to break the cycle.

Practical Deadlock Handling

Remark - Why Many Real Systems Just Ignore It

Many general-purpose operating systems (including most mainstream ones) do not implement formal deadlock avoidance or detection at all, on the practical grounds that deadlocks involving OS-managed resources are comparatively rare in typical workloads - instead relying on application-level discipline (such as consistent lock ordering) and, if a deadlock does occur, simple detection via timeouts or manual intervention (e.g., killing a hung process) rather than the more theoretically complete but costlier algorithms above.
