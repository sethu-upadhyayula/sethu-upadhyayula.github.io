3. Process Synchronization

When multiple threads (Chapter 1) share data, uncoordinated concurrent access can corrupt that data in ways that are difficult to reproduce and debug - this chapter covers the core mechanisms operating systems provide to prevent this.

The Race Condition

Definition - Race Condition

A race condition occurs when the correctness of a computation depends on the relative timing of concurrent operations - for example, two threads both reading a shared counter, incrementing it, and writing it back can lose an update if their operations interleave incorrectly, since neither thread's read-modify-write sequence is guaranteed to complete atomically without explicit protection.

The Critical Section Problem

The region of code that accesses shared data and must not be executed by more than one thread at a time is called a critical section. A correct solution to the critical section problem must guarantee mutual exclusion (only one thread in the critical section at a time), progress (a thread wanting to enter is not blocked indefinitely by threads outside the critical section), and bounded waiting (a limit on how many times other threads can enter before a waiting thread gets its turn).

Locks and Mutexes

Definition - Mutex

A mutex (mutual exclusion lock) is the most basic synchronization primitive - a thread must acquire the mutex before entering a critical section and release it after leaving, with the OS or hardware ensuring only one thread can hold the lock at any moment; any other thread attempting to acquire it blocks until it is released.

Semaphores

Definition - Semaphore

A semaphore is a more general synchronization primitive - an integer counter, adjusted only through atomic increment (signal/V) and decrement (wait/P) operations, that blocks a thread attempting to decrement below zero. A binary semaphore (restricted to values 0 and 1) functions like a mutex, while a general (counting) semaphore can coordinate access to a pool of several identical resources.

Monitors and Condition Variables

A monitor packages a mutex together with the shared data it protects and a set of procedures that are the only way to access that data, automatically ensuring mutual exclusion without the programmer needing to manually acquire and release a lock at every access point. Condition variables let a thread inside a monitor wait for a specific condition to become true, releasing the monitor's lock while waiting and reacquiring it once signaled - the standard mechanism for producer-consumer style coordination.

The Producer-Consumer Problem

Example - Producer-Consumer with a Bounded Buffer

A classic synchronization exercise: producer threads add items to a fixed-size shared buffer, consumer threads remove them, and both must block appropriately when the buffer is full (producers) or empty (consumers) - solved cleanly using a mutex for buffer access plus two counting semaphores (or condition variables) tracking the number of full and empty slots.
