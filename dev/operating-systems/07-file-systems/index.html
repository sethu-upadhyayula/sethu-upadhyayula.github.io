7. File Systems

While Chapters 5–6 covered how the OS manages volatile memory, this chapter covers how it organizes persistent storage - presenting the raw blocks of a disk as the familiar abstraction of files and directories.

Files and Directories

Definition - File and Directory

A file is a named, persistent sequence of bytes, along with metadata (size, permissions, timestamps) the OS tracks about it. A directory is itself a special kind of file containing a mapping from names to other files or directories, allowing the hierarchical file system structure most operating systems present to users.

File Allocation Methods

How a File's Blocks Are Tracked on Disk
  • Contiguous allocation - a file occupies a single run of consecutive disk blocks; fast to read sequentially, but suffers external fragmentation and requires knowing a file's final size in advance.
  • Linked allocation - each block stores a pointer to the next block of the same file; eliminates fragmentation, but performs poorly for random access since reaching a given block requires following the chain from the start.
  • Indexed allocation - a dedicated index block stores pointers to all of a file's data blocks, supporting efficient random access without contiguous allocation's fragmentation problem; the basis for most modern file systems' block-mapping structures.

Inodes

Definition - Inode

In Unix-family file systems, an inode is the per-file metadata and block-index structure described above, identified by a number rather than directly by its file name - a directory entry simply maps a human-readable name to an inode number, which is why a file can have multiple names (hard links) pointing to the exact same underlying inode and data.

Free Space Management

The file system must also track which disk blocks are currently free versus allocated - commonly via a bitmap (one bit per block) or a free list (a linked list of free blocks) - consulted whenever a file needs to grow or a new file is created and blocks must be allocated to it.

Journaling and Crash Consistency

Remark - Surviving a Crash Mid-Write

A system crash or power loss partway through a multi-step file system update (e.g., updating both a file's data and its metadata) can leave the file system in an inconsistent state. Journaling file systems address this by first writing a description of an intended update to a separate log (journal) before applying it to the main file system structures - after a crash, the journal can be replayed to complete or safely roll back any update that was interrupted, restoring consistency without requiring a full, slow scan of the entire disk.
