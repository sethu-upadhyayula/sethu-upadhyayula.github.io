1. Processes and Threads

The process is the operating system's fundamental abstraction for a running program - this chapter covers what a process actually consists of, and how threads let a single process do more than one thing at once.

What a Process Is

Definition - Process

A process is a running instance of a program, consisting of its own address space (Chapter 5), a set of registers and program counter describing its current execution state, open file descriptors, and other OS-managed resources. Two processes running the identical program code are still entirely distinct processes, each with its own private memory and state.

The Process Control Block

The OS tracks every process's state in a process control block (PCB) - a kernel data structure recording the process's ID, current state (running, ready, blocked), saved register values, memory management information, and scheduling priority (Chapter 2) - consulted and updated every time the OS switches which process is actually running on the CPU.

Process States and Transitions

The Basic Process State Diagram

A process moves between states over its lifetime: new (being created), ready (waiting for CPU time), running (currently executing), waiting/blocked (waiting on I/O or another event), and terminated. The scheduler (Chapter 2) governs transitions between ready and running, while I/O completion or event signaling governs transitions out of the waiting state.

Threads: Sharing a Process's Resources

Definition - Thread

A thread is an independent sequence of execution within a process, with its own program counter, register set, and stack, but sharing the process's address space and open files with any other threads in that same process. Because threads share memory directly, communication between them is cheap (no OS-mediated message passing required) but requires careful synchronization (Chapter 3) to avoid corrupting shared data.

Context Switching

Switching the CPU from running one process or thread to another - a context switch - requires saving the current thread's registers and program counter to its PCB and loading the next thread's saved state, a pure overhead cost (no useful work is done during the switch itself) that the scheduler (Chapter 2) must weigh against the benefits of frequent switching, such as responsiveness.

Inter-Process Communication

Because separate processes do not share memory by default, communicating between them requires explicit OS-provided mechanisms - pipes, message queues, shared memory segments (which processes must explicitly opt into), and sockets (Chapter 1 of Computer Networks) - each with different tradeoffs in speed, complexity, and whether communication is restricted to processes on the same machine.
