8. Interpreters, JIT Compilation, and Runtime Systems

This closing chapter covers alternatives and complements to the ahead-of-time compilation pipeline covered in Chapters 1–7 — interpretation, just-in-time compilation, and the runtime support that executing code, however it was produced, ultimately depends on.

Interpreters

Definition — Interpreter

An interpreter executes a program directly — often from a simple intermediate bytecode form rather than raw source text — without first translating it into native machine instructions for the target processor. A bytecode interpreter repeatedly fetches, decodes, and executes bytecode instructions in a loop, conceptually similar to the fetch-decode-execute cycle of a physical processor (Chapter 2 of Computer Architecture), but implemented in software.

Just-In-Time Compilation

Definition — JIT Compilation

Just-in-time (JIT) compilation compiles code to native machine instructions at run time, rather than entirely ahead of time — often selectively, compiling only the "hot" portions of code actually executed frequently (identified through runtime profiling), while less frequently executed code may remain interpreted — combining much of ahead-of-time compilation's execution speed with an interpreter's flexibility and fast startup time, at the cost of runtime compilation overhead and implementation complexity.

Adaptive Optimization

Because a JIT compiler observes a program's actual runtime behaviour, it can apply optimizations an ahead-of-time compiler cannot — speculative optimization based on commonly observed runtime types or values, with a fallback ("deoptimization") path if that speculation later turns out to be wrong for some specific execution — a genuinely more information-rich optimization opportunity than static, ahead-of-time compilation (Chapter 6) has access to, since the ahead-of-time compiler must remain correct for every possible input, not just the commonly observed one.

Garbage Collection

Definition — Garbage Collection

Many language runtimes provide automatic memory management (garbage collection), reclaiming memory occupied by objects no longer reachable from any live reference, rather than requiring the programmer to explicitly free memory (as in C). Common approaches include reference counting (tracking how many references point to each object, freeing it once the count reaches zero) and tracing garbage collection (periodically identifying all objects still reachable from a set of known roots, and reclaiming everything else) — each with different tradeoffs in overhead, pause times, and the ability to collect reference cycles.

The Runtime System

Remark — What "Runtime" Actually Includes

Beyond garbage collection, a language's runtime system typically also handles thread and stack management (Chapter 1 of Operating Systems), exception handling and stack unwinding, and dynamic type information needed for features like runtime type checks — infrastructure that compiled (or interpreted) code calls into constantly during execution, meaning a language's true performance and behaviour characteristics depend as much on its runtime system's design as on the compiler pipeline covered in the rest of this subject.
