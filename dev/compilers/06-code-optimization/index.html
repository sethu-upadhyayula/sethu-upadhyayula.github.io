6. Code Optimization

Once a program is translated into an intermediate representation (Chapter 5), a compiler can apply transformations that improve its performance — without changing the program's observable behaviour at all — before final code generation (Chapter 7).

Local versus Global Optimization

Local optimizations operate within a single basic block (Chapter 5), requiring no cross-block analysis. Global (intraprocedural) optimizations reason across an entire function's control flow graph, and interprocedural optimizations go further still, reasoning across function call boundaries — each successive scope enables more powerful transformations, at the cost of more expensive analysis to perform them correctly.

Common Optimization Techniques

Frequently Applied Optimizations
  • Constant folding — evaluating constant expressions (e.g., 2 + 3) at compile time rather than at run time.
  • Common subexpression elimination — computing a repeated expression's value once and reusing it, rather than recomputing it every time it appears.
  • Dead code elimination — removing computations whose results are never subsequently used.
  • Loop-invariant code motion — moving a computation that produces the same result on every loop iteration out of the loop entirely.
  • Function inlining — replacing a function call with the function's actual body, eliminating call overhead and often exposing further optimization opportunities at the call site.

Data Flow Analysis

Definition — Data Flow Analysis

Data flow analysis computes, for each point in a program's control flow graph (Chapter 5), some property of interest about the program's runtime behaviour — such as which variable definitions might still be "live" (used later) at a given point — by propagating information along the graph's edges until reaching a stable fixed point. Many optimizations (dead code elimination above, for instance) depend directly on the results of a specific data flow analysis to determine safely where they can be applied.

Correctness: Preserving Program Meaning

Remark — The Overriding Constraint on Any Optimization

Every optimization must preserve the program's observable behaviour exactly — an optimization that changes a program's result, even in a rare edge case, is not a valid optimization at all but a compiler bug. This constraint is what separates legitimate optimization (however aggressive) from simply generating different, faster-but-wrong code, and is why optimizations are typically proven correct (or at least carefully justified) against the language's formal or de facto semantics before being included in a production compiler.

Optimization Levels

Compilers typically expose configurable optimization levels (such as -O0 through -O3 in many C/C++ compilers), trading compilation time and debuggability against runtime performance — lower levels compile faster and preserve a closer correspondence between source code and generated instructions (easing debugging), while higher levels apply progressively more aggressive transformations in pursuit of faster generated code.
