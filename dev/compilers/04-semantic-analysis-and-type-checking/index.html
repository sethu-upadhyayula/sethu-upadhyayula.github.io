4. Semantic Analysis and Type Checking

A program can be perfectly well-formed syntactically (Chapter 3) and still be meaningless or invalid — using an undeclared variable, or adding a number to a string in a language that forbids it. Semantic analysis catches exactly these problems.

Scope and Declaration Checking

Semantic analysis verifies that every identifier used in a program was actually declared, in a scope where that declaration is visible at the point of use — consulting and building the symbol table introduced in Chapter 1 to track which declarations are visible at each point in the program, correctly handling nested scopes (a variable declared inside a block shadowing one of the same name from an enclosing scope).

Type Checking

Definition — Type Checking

Type checking verifies that every operation in a program is applied to operands of compatible types — assigning a type to each expression in the abstract syntax tree (Chapter 3) and confirming that types match according to the language's typing rules (e.g., that an argument passed to a function matches the parameter's declared type), rejecting the program with a compile-time type error if any check fails.

Static versus Dynamic Typing

Definition — Static and Dynamic Typing

A statically typed language performs type checking at compile time, catching type errors before the program ever runs. A dynamically typed language instead defers type checking to run time, checking a value's actual type only when an operation is performed on it — trading the earlier error detection static typing provides for greater flexibility (and, often, less verbose code), a tradeoff different languages resolve differently, and one relevant to how a language's interpreter or JIT compiler (Chapter 8) is structured.

Type Inference

Type inference allows a statically typed language to determine an expression's type automatically from context, without requiring the programmer to write an explicit type annotation everywhere — reducing the verbosity typically associated with static typing while retaining its compile-time error-catching benefit, at the cost of type-checking algorithms that are often considerably more sophisticated than simple explicit-annotation checking alone.

Other Semantic Checks

Further Semantic Analysis Tasks
  • Verifying a function is called with the correct number of arguments.
  • Resolving operator and function overloading, choosing the correct specific implementation based on argument types.
  • Checking control-flow-related requirements, such as ensuring every path through a non-void function returns a value.
