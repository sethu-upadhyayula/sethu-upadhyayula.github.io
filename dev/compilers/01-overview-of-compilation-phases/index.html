1. Overview of Compilation Phases

Translating source code into executable machine instructions is too large a task to perform in a single step — this chapter introduces the standard sequence of phases a compiler breaks that translation into, each covered in depth in a later chapter of this subject.

The Front End, Middle End, and Back End

The Three Broad Stages
  • Front end — lexical analysis (Chapter 2), parsing (Chapter 3), and semantic analysis (Chapter 4); understands the source language and checks the program's validity.
  • Middle end — operates on an intermediate representation (Chapter 5), performing target-independent optimizations (Chapter 6).
  • Back end — code generation (Chapter 7), translating the (optimized) intermediate representation into instructions for a specific target architecture (Computer Architecture).

Why Separate Front End from Back End

Remark — The Value of This Separation

Structuring a compiler around a shared intermediate representation (Chapter 5) that decouples the front end from the back end means a single front end (for one source language) can target multiple back ends (different processor architectures), and a single back end can serve multiple front ends (different source languages) — avoiding the need to write a completely separate compiler for every combination of source language and target architecture.

Compilation versus Interpretation

A compiler translates source code into a different, typically lower-level form (machine code, or an intermediate bytecode) before execution, while an interpreter directly executes source code (or a simple intermediate form) statement by statement, without a separate full translation step — a distinction covered further, along with hybrid approaches, in Chapter 8.

The Symbol Table

A symbol table is a data structure maintained throughout much of the compilation pipeline, recording every identifier (variable, function, type) declared in a program along with its scope, type, and other attributes — consulted and updated across parsing (Chapter 3), semantic analysis (Chapter 4), and code generation (Chapter 7), serving as shared, persistent bookkeeping across otherwise fairly independent compilation phases.

Error Handling Across Phases

Each compilation phase can, in principle, detect errors specific to its own concern — a lexical error (Chapter 2, an invalid character sequence), a syntax error (Chapter 3, tokens not forming a valid program structure), or a semantic error (Chapter 4, a type mismatch) — with a well-engineered compiler aiming to report as many genuine errors as possible in a single pass rather than stopping at the very first one encountered, to make the edit-compile-fix cycle covered in Chapter 7 of Software Engineering & Systems Design more efficient for the programmer.
