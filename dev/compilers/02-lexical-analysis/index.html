2. Lexical Analysis

The first phase of compilation converts a program's raw character stream into a sequence of meaningful units — tokens — that the rest of the compiler can work with, rather than dealing with individual characters directly.

Tokens

Definition — Token

A token is a categorized, meaningful unit of source code — a keyword, identifier, numeric literal, operator, or punctuation symbol — produced by the lexer (or scanner) from the raw character stream, discarding irrelevant details such as whitespace and comments along the way (though comments and whitespace can matter for some purposes, such as language-aware tooling, even if they're irrelevant to the language's actual semantics).

Regular Expressions and Lexer Specification

Token patterns are typically specified using regular expressions — for instance, an identifier might be specified as a letter followed by any number of letters or digits — with lexer-generator tools (such as Lex/Flex) compiling a set of such regular expressions directly into an efficient scanning program, rather than requiring the compiler author to hand-write character-by-character scanning logic.

Finite Automata

Definition — DFA-Based Scanning

Regular expressions are converted (via a nondeterministic finite automaton, then determinized) into a deterministic finite automaton (DFA) that recognizes a given token pattern by transitioning between states one input character at a time, accepting the input once it reaches a designated final state — providing a scanning implementation whose running time is linear in the length of the input, with no backtracking required.

Maximal Munch

When more than one token pattern could match at a given point (for instance, deciding whether <= should scan as a single "less-than-or-equal" token or as two separate tokens "<" and "="), lexers conventionally apply the maximal munch rule — always preferring the longest possible match — resolving this ambiguity consistently without requiring lookahead into later tokens.

Lexical Errors

Remark — Handling Invalid Input

A character sequence matching no valid token pattern at all (such as an unterminated string literal, or a genuinely invalid character in the source language) constitutes a lexical error — typically the earliest kind of error a compiler can detect, and one many compilers attempt to recover from gracefully (e.g., skipping the offending characters and resuming scanning) so that later, independent errors elsewhere in the file can also be reported in the same compilation pass.
