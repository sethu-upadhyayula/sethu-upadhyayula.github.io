3. Parsing

Given the token stream produced by lexical analysis (Chapter 2), the parser organizes those tokens into a tree structure reflecting the program's grammatical structure according to the language's formal grammar.

Context-Free Grammars

Definition — Context-Free Grammar

A programming language's syntax is typically specified as a context-free grammar — a set of production rules, each rewriting a single non-terminal symbol into a sequence of terminals (tokens) and/or other non-terminals — precise enough to define exactly which token sequences form valid programs, and structured enough that efficient parsing algorithms exist for well-behaved subsets of context-free grammars.

Parse Trees and Abstract Syntax Trees

A parse tree directly reflects every grammar rule applied during parsing, including grammatical details (such as explicit grouping parentheses) with no bearing on the program's actual meaning. An abstract syntax tree (AST) is a simplified version, retaining only the structure relevant to semantic meaning — the representation the rest of the compiler (Chapters 4–7) actually works with going forward.

Top-Down Parsing

Definition — Recursive Descent Parsing

A top-down parser builds a parse tree starting from the grammar's root symbol, working down to match the actual input tokens. Recursive descent is the most common hand-written top-down approach, implementing one recursive function per grammar non-terminal — simple to write and understand, but requiring the grammar to avoid left recursion (a rule that refers to itself as its own first symbol) to avoid infinite recursive descent.

Bottom-Up Parsing

Definition — LR Parsing

A bottom-up parser instead starts from the input tokens and progressively combines them upward into larger grammatical structures until reaching the grammar's root symbol. LR parsing is the standard bottom-up technique, using a stack and a precomputed parsing table (typically generated automatically by a parser-generator tool, such as Yacc/Bison) to decide, at each step, whether to shift the next token onto the stack or reduce a sequence of stack symbols according to a grammar rule.

Ambiguity and Precedence

Remark — Resolving Grammatical Ambiguity

A grammar is ambiguous if some valid input can be parsed in more than one structurally different way (the classic example being a + b * c, ambiguous without an explicit precedence rule for whether multiplication binds tighter than addition) — resolved either by rewriting the grammar to encode precedence and associativity directly and unambiguously, or by supplying explicit precedence declarations that a parser generator uses to resolve shift/reduce conflicts automatically.

Syntax Errors

A token sequence not matching any valid grammar derivation constitutes a syntax error — a well-engineered parser aims not just to detect such errors but to recover from them gracefully (for instance, skipping ahead to a recognizable synchronization point, such as the next statement boundary) so it can continue parsing and report further, independent errors in the same file rather than stopping at the very first one.
